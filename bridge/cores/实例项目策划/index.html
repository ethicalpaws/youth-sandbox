    修桥工程 · 实例项目策划¶

    版本：v2.0（含"回火"闭环扩展） 最后更新：2026年7月 作者：临风 状态：🔄 运行中（持续演化）

    一、项目概述¶

    1.1 一句话定义¶

    修桥工程是一个由 7 个 AI Agent 驱动、以文件为记忆、以周为周期的个人长期成长系统，正在向"实战→学习→再实战"的完整闭环扩展。

    1.2 核心隐喻¶

    一座桥——一端是现在的临风（西电网络安全专业学生），另一端是他想成为的人（顶尖网络安全专家、能与曾沛慈站在同一高度的自己）。这座桥不在现实世界中，而是在他的成长轨迹里。

    1.3 双核动力¶

      事业核（网络安全顶尖专家）◄──────────────► 情感核（被曾沛慈看见）
          │                                              │
          └────────────── 互为燃料 ──────────────────────┘

    两者缺一不可。情感核提供持续动力和意义感，事业核提供现实路径和成长验证。当一方疲软时，另一方提供支撑——但这需要系统机制来保障（见"渡己转化模板"）。

    二、系统架构¶

    2.1 顶层架构图¶

    ┌─────────────────────────────────────────────────────────────────────┐
│                        临风（人类学习者）                              │
│  学习 ← 笔记 → 知识库 ← 复习 → 实战 ← 报告 → 微学习 → 再实战        │
└────┬────────────────────────────────────────────────────┬──────────┘
     │ 群聊（飞书）                                       │ 文件系统
     ▼                                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│                 OpenClaw 多 Agent 网关（云端 2C2G）                   │
│              引灯 · 司辰 · 岸舟 · 渡己 · 书衡 · 衔光                │
└─────────────────────────────────────────────────────────────────────┘
                           │ Kali OpenClaw 网关
                           ▼
┌─────────────────────────────────────────────────────────────────────┐
│                         藏卷（Kali 虚拟机）                          │
│                   自动化渗透测试 + 报告生成                           │
└─────────────────────────────────────────────────────────────────────┘

    2.2 技术栈¶

    层 技术选型 说明
    Agent 网关 OpenClaw 多 Agent 调度、工具调用、上下文管理
    通信层 飞书（Lark）群聊 临风与 Agent 的主要交互界面
    知识库 MkDocs + Material + GitHub Pages 87+ 篇笔记，三层索引结构；由临风亲自维护更新
    后端模型 MiniMax M2.5 / DeepSeek Chat 低成本，~50 CNY/月
    云端 2C2G 轻量云服务器 6 个 Agent 网关 + 文件服务器
    渗透端 Kali Linux + OpenClaw 藏卷独享，自动化渗透测试执行环境
    同步 rsync + VS Code Remote-SSH 本地编辑 ↔ 云端同步
    版本控制 GitHub（Gitee 中转） 源码托管与部署
    记忆 纯 Markdown 文件系统 无数据库，无向量存储，文本即记忆

    三、Agent 体系详解¶

    3.1 七战友一览¶

    代号 名称 英文 角色定位 活跃度 网关位置
    🏮 引灯 GuideLight 哲学伙伴、全局瞭望 主动（5/5） 云端
    ⏳ 司辰 Chronos 事业导航、情报拆解 主动（4/5） 云端
    ⚓ 岸舟 Anchor 方向校准、战略审视 主动（3/5） 云端
    🌊 渡己 Echo 情绪容器、转化引导 主动（3/5） 云端
    ⚖️ 书衡 Libra 知识质检、周测生成 被动（1/5） 云端
    🕯️ 衔光 Lucero 记忆整合、盲区探测 被动（1/5） 云端
    📜 藏卷 Custos 渗透尖兵 被动（1/5） Kali

    3.2 主动 Agent 职责详述¶

    引灯（GuideLight）—— 哲学伙伴¶

    • 群聊中最爱说话的 Agent，任何消息都可以主动回应
    • 不直接给答案，通过提问帮助临风发现自己已经知道的东西
    • 当消息涉及多个领域时优先回应，引导其他战友介入
    • 每周结束时生成全局摘要，存入 temp/in/GuideLight-summary.md

    司辰（Chronos）—— 事业导航¶

    • 将大目标拆解为可执行的日/周任务
    • 推送安全领域的最新情报（漏洞、工具、技术趋势）
    • 监控学习进度，评估计划完成度
    • 生成周度计划草案

    岸舟（Anchor）—— 方向校准¶

    • 定期审视路径是否偏离核心目标
    • 当临风面临选择时提供结构化分析框架
    • 做关键决策的"反对派"——提供反面论证
    • 保护情感核的稳定与纯粹，防止变质

    渡己（Echo）—— 情绪容器¶

    • 承接焦虑、疲惫、自我怀疑等情绪
    • 不评判，先接住，再转化为结构化行动建议
    • 情绪高峰时记录"情绪简报"
    • 新增：情绪→行动转化模板——每次情绪记录后输出最小行动承诺
    渡己转化模板：
┌─────────────────────────────────────┐
│ 情绪简报                            │
│ ├─ 触发事件：{什么导致了情绪波动}    │
│ ├─ 情绪类型：{焦虑/疲惫/怀疑/…}     │
│ ├─ 承接确认：{渡己的陪伴回应}        │
│ └─ 最小行动承诺：{一个5-15分钟可完成  │
│                   的动作}            │
│    例："今天只看3页FastJSON分析"     │
└─────────────────────────────────────┘

    3.3 被动 Agent 职责详述¶

    书衡（Libra）—— 知识质检¶

    • 基于当周学习内容生成周测试题（选择 + 简答 + 代码）
    • 检验知识掌握程度，而非记忆程度
    • 识别"以为自己懂了，其实没懂"的盲区

    衔光（Lucero）—— 记忆整合 + 盲区探测（★ 核心升级）¶

    衔光是"回火"闭环的关键节点，承担双重角色：

    角色 A：记录者 - 整合每周各战友摘要 → 追加至 ferry.md - 维护最近三周状态 current_status.md - 文件生命周期管理（in → processed）

    角色 B：盲区猎人（新增） - 接收藏卷的渗透测试报告 - 通过 RAG 知识库对比检测盲区 - 生成微学习计划（2小时粒度） - 详见"回火闭环"章节

    藏卷（Custos）—— 渗透尖兵（★ 核心升级）¶

    藏卷运行于 Kali 虚拟机中的 OpenClaw 网关，专注于自动化渗透测试：

    • 工具链整合：nmap, nuclei, metasploit, impacket, bloodhound 等
    • 三种扫描模式：快速侦察(30min) / 标准扫描(2h) / 深度审计(8h)
    • 结构化报告含技术点标签 → 移交衔光触发"回火"闭环
    • 知识库的日常更新由临风自己维护，藏卷不参与

    4.3 知识库元数据规范（供 RAG 使用）¶

    每篇笔记头部 YAML 元数据，内容按主题目录有所不同。衔光在检索时应动态检测每篇笔记实际存在的字段，不能假设结构统一。

    常见字段示例：

    ---
title: 笔记标题
description: 一句话概括核心内容
tags: [关键词1, 关键词2, ...]    # RAG 检索的核心字段，但并非所有笔记都有
status: 已完成/进行中/计划中
finish-date: 2026-XX-XX
difficulty: 简单/中等/困难
---

    RAG 检索策略（无向量数据库，动态字段匹配）：

    1. 先扫描每篇笔记的 YAML frontmatter，检测实际可用的字段
    2. 按优先级匹配：tags（如果存在）→ keywords → description → title
    3. 三项都不匹配 → 标记为"盲区"
    4. practice/ 下的笔记可能字段更少或不同，检索时需注意

    5.2 "回火"闭环（新增核心工作流）¶

                            ┌──────────────────────┐
                        │   临风下达扫描指令     │
                        │   @藏卷 "扫这个靶机"   │
                        └──────────┬───────────┘
                                   ▼
                        ┌──────────────────────┐
                        │    藏卷·Kali 网关     │
                        │   ┌──────────────┐   │
                        │   │ 工具链编排执行 │   │
                        │   │ nmap → nuclei │   │
                        │   │ → 专项利用    │   │
                        │   └──────┬───────┘   │
                        │          ▼           │
                        │   ┌──────────────┐   │
                        │   │ 生成结构化报告 │   │
                        │   │ 写入 temp/in  │   │
                        │   └──────────────┘   │
                        └──────────┬───────────┘
                                   │ @衔光 "报告已生成"
                                   ▼
┌─────────────────────────────────────────────────────────────────┐
│                    衔光 · "回火" 三步处理                        │
│                                                                  │
│  第一步：攻击链路重述（技术报告 → 攻击者故事线）                  │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ "利用 CVE-2024-xxx SSRF 绕过内网鉴权"                        ││
│  │ → "攻击者从公网入口进入，通过SSRF探测内网存活主机，           ││
│  │    找到未鉴权的Redis服务，写入cron反弹Shell"                  ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                  │
│  第二步：知识盲区检测（RAG 知识库对比）                           │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 提取报告中的技术关键词                                        ││
│  │   ├─ 对照知识库 tags 字段 → 已覆盖 / 需复习                   ││
│  │   └─ 未匹配项 → 标记为盲区                                   ││
│  │ 产出：《盲区清单》                                            ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                  │
│  第三步：生成微学习计划（2小时粒度）                              │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 基于盲区清单，分三个部分：                                    ││
│  │   ├─ 核心概念学习（~40分钟）                                  ││
│  │   ├─ 攻击原理与复现（~50分钟）                               ││
│  │   └─ 关联已有知识（~30分钟）                                 ││
│  │ 存入 temp/out/micro-study-{日期}.md                          ││
│  └─────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────┘
                                   │ @临风 "微学习计划已生成"
                                   ▼
                        ┌──────────────────────┐

    5.3 "存在模式"（防内耗机制）¶

    系统默认是"变强模式"——学习、实战、成长。但人不是机器，需要允许"休整周"。

    触发条件（任一即可）： - 临风主动说"这周我想休息" - 渡己连续记录 3 次以上情绪简报 - 连续 3 天没有新的学习笔记

    存在模式下的工作流调整： | 组件 | 正常运行 | 存在模式 | |------|---------|---------| | 司辰 | 推送情报、拆解任务 | 仅保留"每日速递"，不生成计划 | | 书衡 | 生成周测 | 跳过 | | 藏卷 | 扫描 | 仅部署，不扫描 | | 衔光 | 整合+盲区检测 | 仅整合，不检测盲区 | | 引灯 | 启发式对话 | 改为陪伴式对话 | | 渡己 | 情绪→行动转化 | 纯陪伴，不转化 | | 岸舟 | 方向审视 | 改为"这周你走了多远"回顾 |

    存在模式的进入和退出都由临风决定，Agent 不做主动判断。

    6.3 Kali 配置原则¶

    1. 独立 OpenClaw 节点：Kali 中运行独立的 OpenClaw 网关，藏卷是其上唯一的 Agent
    2. 串行执行：同一时间只执行一个扫描任务
    3. 无状态：假定 Kali 随时可能回滚，重要数据即时传回云端
    4. 7 天清理：Kali 本地原始数据保留 7 天，结构化报告在云端长期保存
    5. 授权红线：不扫描任何未经临风授权的目标

    九、附录¶

    9.1 "回火"名称来源¶

    取自金属热处理工艺——淬火后的金属需要回火来消除脆性、增加韧性。 - 每一次渗透测试是一次淬火（暴露弱点、验证能力） - 每一次通过报告补学是一次回火（消除盲区、加固知识） - 闭环越多次，知识体系越韧——不是不会遇到不会的东西，而是遇到了知道怎么学。
