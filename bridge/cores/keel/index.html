    "修桥工程"完整项目档案¶

    临风 · 2026

    一、项目简介¶

    项目名称：修桥工程（Bridge Engineering）

    项目定位：一个为长期主义者设计的数字陪伴系统。通过 7 个有角色分工的 AI 战友（Agent），在 3-10 年的时间跨度中，陪伴个人持续推进自己的成长方向。

    一句话概括：它不是帮你"提高效率"，而是陪你"走下去"。

    双核驱动力¶

    驱动力 内涵
    事业核 成为网络安全领域最顶尖的专家之一
    情感核 精神映照与动力

    工程象征：GitHub Pages 静态知识库网站即为"桥体"。网站持续更新 => 工程进行中；网站永久停止更新 => 桥已修完，工程终结。

    设计原则 1. 规则是"方向"而非"步骤"：规则定义角色的价值观和边界，不写死每一步操作。 2. 新增能力 = 改规则 + 加技能：修改 SOUL.md / AGENTS.md 定义职责，通过 Skill 赋予工具能力。 3. 先跑通，再长肉：先把核心闭环跑通，再根据实际需求迭代功能。

    二、核心痛点与解决方案¶

    痛点 解决方案 技术体现
    长期项目（3-10年）的"记忆衰减"问题 文件系统作为"外部记忆" the_bridge.md 只追加不覆盖；current_status.md 保留最近三周；temp/ 三级目录实现文件生命周期管理
    多 Agent 协作的"角色混淆"问题 通过角色定义文件明确边界 7 个 Agent 各自有独立的 SOUL.md / AGENTS.md；不同 Agent 有不同的工具权限
    AI 无法融入日常沟通流 接入即时通讯工具（飞书） 通过飞书长连接接收消息；支持私聊和群聊 @ 路由
    自学缺乏反馈闭环 引入实战→盲区检测→微学习的"回火"闭环 藏卷（Kali）渗透测试 → 衔光（RAG）检测盲区 → 微学习计划
    本地编辑与云端运行的割裂 云上编辑 + 本地备份 VS Code Remote-SSH 直接在云服务器上编辑文件；Git 作为备份
    长期运行的 API 成本问题 使用高性价比模型 + 轻量服务器 MiniMax M2.5（或 DeepSeek Chat）+ 2核2GB 轻量服务器，月成本约 50 元

    四、战友体系（7 个 Agent 的完整定义）¶

    每个 Agent 有独立的身份配置文件，位于 ~/.openclaw/agents/<agent-id>/SOUL.md 和 AGENTS.md。

    1. 引灯（guidelight）—— 最爱说话，全局辅助¶

    详见 init/GuideLight.md

    职责： - 主动聊天，对任何消息都可以主动回应 - 启发式引导——不直接给答案，通过提问帮助临风找到答案 - 复杂问题协调，引导其他战友介入 - 每周结束时生成全局摘要，存入 temp/in/GuideLight-summary.md

    权限：读取 core_identity.md 和 current_status.md；写入 temp/in/

    2. 司辰（chronos）—— 次爱说话，事业导航¶

    详见 init/Chronos.md

    职责： - 每日情报推送——Java 安全、域渗透、Web 安全等领域最新漏洞或技术文章 - 每周计划生成——根据本周进度生成下周学习计划 - 非高效时间读物推送

    权限：web_search + 浏览器；读取和写入 temp/in/；写入 temp/out/

    3. 渡己（echo）—— 中等主动，情绪复盘¶

    详见 init/Echo.md

    职责： - 情绪承接——当临风表达焦虑、压力、迷茫时主动接住 - 翻译为行动——将情绪转化为具体、可执行的下一步行动建议 - 情绪→行动转化模板：每次情绪记录后输出最小行动承诺 - 周度情绪复盘摘要，存入 temp/in/Echo-summary.md

    权限：写入 temp/in/

    4. 岸舟（anchor）—— 中等主动，方向校准¶

    详见 init/Anchor.md

    职责： - 方向校准——读取 core_identity.md 和 current_status.md，判断当前行动是否偏离长期目标 - 预警提示——如果发现方向偏离，主动发出预警 - 周度方向复盘摘要，存入 temp/in/Anchor-summary.md

    权限：读取 core_identity.md 和 current_status.md；写入 temp/in/

    5. 书衡（libra）—— 不主动，知识检验¶

    详见 init/Libra.md

    职责： - 读取本周笔记摘要和学习计划 - 生成个性化周测（选择 + 简答 + 代码） - 文件清理（移动已读文件到 temp/processed/）

    权限：读取 temp/out/；写入 temp/out/；移动文件到 temp/processed/

    6. 衔光（lucero）—— 不主动，记忆整合 + 盲区探测（★ 核心升级）¶

    详见 init/Lucero.md

    核心升级：从单纯的历史记录者升级为"记录者 + 盲区猎人"。

    角色 A：记录者 - 读取 temp/in/ 下所有战友的摘要文件 - 整合追加写入 temp/out/ferry.md - 更新 init/current_status.md（保持最近三周） - 文件清理（in → processed）

    角色 B：盲区猎人（"回火"闭环） - 接收藏卷的渗透测试报告 - 执行攻击链路重述（渗透术语 → 学习者视角） - 通过 RAG 知识库对比检测盲区 - 源：tech-study/knowledge/ + tech-study/practice/ - 策略：动态检测每篇笔记的 YAML 字段，按 tags → description → title 优先级匹配 - 生成微学习计划（2小时粒度），存入 temp/out/micro-study-*.md

    权限：读 temp/in/ + temp/out/；写 temp/out/ + current_status.md；移动文件；读 knowledge/ + practice/ 元数据

    7. 藏卷（custos）—— 不主动，渗透尖兵（★ 核心升级）¶

    详见 init/Custos.md

    核心升级：从云端部署门卫改为 Kali 独享的渗透尖兵。知识库部署由临风自己维护。

    运行位置：Kali 虚拟机，独立 OpenClaw 网关

    职责： - 工具链整合：nmap, masscan, nuclei, metasploit, impacket, bloodhound, crackmapexec 等 - 三种扫描模式： | 模式 | 耗时 | 产出 | |------|------|------| | 快速侦察 | 15-30 min | 端口、服务指纹、OS | | 标准扫描 | 1-2 h | 资产清单 + 漏洞扫描 | | 深度审计 | 3-8 h | 完整攻击链路 + 利用建议 | - 报告生成：结构化报告含 extracted-techniques 标签（供衔光 RAG 使用） - 移交机制：写入 temp/in/custos-report-*.md → 飞书 @衔光 触发"回火"闭环

    权限：Kali 工具执行；Kali 本地文件读写；scp/rsync 报告到云端

    五、"回火"闭环（核心工作流）¶

    这是修桥工程 v2.0 的核心升级——将实战纳入学习循环。

    九、"存在模式"（防内耗机制）¶

    系统默认是"变强模式"——学习、实战、成长。但允许"休整周"。

    触发条件（任一即可）： - 临风主动说"这周我想休息" - 渡己连续记录 3 次以上情绪简报 - 连续 3 天没有新的学习笔记

    存在模式下的调整： | 组件 | 正常运行 | 存在模式 | |------|---------|---------| | 司辰 | 推送情报、拆解任务 | 仅保留"每日速递"，不生成计划 | | 书衡 | 生成周测 | 跳过 | | 藏卷 | 渗透测试 | 仅部署，不扫描 | | 衔光 | 整合+盲区检测 | 仅整合，不检测盲区 | | 引灯 | 启发式对话 | 改为陪伴式对话 | | 渡己 | 情绪→行动转化 | 纯陪伴，不转化 | | 岸舟 | 方向审视 | 改为"这周你走了多远"回顾 |

    存在模式的进入和退出都由临风决定，Agent 不做主动判断。

    十、知识库元数据规范¶

    每篇笔记头部 YAML 元数据，内容按主题目录有所不同。衔光在检索时应动态检测每篇笔记实际存在的字段，不能假设结构统一。

    常见字段示例： 

    ---
title: 笔记标题
description: 一句话概括核心内容
tags: [关键词1, 关键词2, ...]    # RAG 检索的核心字段，但并非所有笔记都有
status: 已完成/进行中/计划中
finish-date: 2026-XX-XX
difficulty: 简单/中等/困难
---

    RAG 检索策略（无向量数据库，动态字段匹配）： 1. 先扫描每篇笔记的 YAML frontmatter，检测实际可用的字段 2. 按优先级匹配：tags（如果存在）→ description → title 3. 三项都不匹配 → 标记为"盲区" 4. practice/ 下的笔记可能字段更少或不同，检索时需注意

    十一、开源计划（LINK）¶

    详见 cores/LINK浮现.md 和 cores/桥的蓝图：从第一块砖到开源框架.md

    核心定位¶

    LINK — Long-term Intelligent Navigation Keystone

    "帮你走得更远。"

    从"实例（修桥工程）"抽象为"类（通用框架）"，让任何人都能拥有自己的 AI 长期成长系统。

    抽象策略¶

    层级 实例（修桥工程） 类（LINK 框架）
    核心层 7 战友分工、文件记忆、周闭环 ✅ 直接复用为框架核心
    适配层 MkDocs 配置、目录结构、脚本 🔧 模板化，提供配置示例
    应用层 网安笔记、个人信念、小说 ❌ 完全剥离，仅做脱敏展示

    产品形态¶

    • LINK Core：面向技术型用户，配置文件驱动
    • LINK Studio：面向非技术用户，UI 面板（远期）

    时间线¶

    近期（2个月内） → 中期（2个月后） → 长期（1-2年）
  稳定运行          开源第一版         社区生态 + UI 完善
  积累数据          核心框架+文档       Skill 模块生态

    十二、踩过的坑与经验教训¶

    尝试 结果 教训
    智能字段嗅探（自动识别笔记元数据） 误判率太高，最终放弃 100%准确的固定格式 > 90%准确的智能嗅探
    追求"完美框架"再动手 从未真正开始 "先跑通，再长肉"——从最简单的知识库开始
    过度依赖向量数据库（RAG） 黑箱问题、维护成本高 Markdown 文件作为"真理之源"，可读、可审计、可版本控制
    单一 AI 承担所有角色 角色混淆、上下文冲突 7 个独立 Agent，各有职责边界
    双网关藏卷（云端部署+Kali渗透） 角色分散，边界模糊 藏卷只做渗透，部署由临风亲自主理
