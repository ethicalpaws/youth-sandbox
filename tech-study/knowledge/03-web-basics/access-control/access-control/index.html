    访问控制漏洞

    什么是访问控制¶

    1. 在web应用中，访问控制决定了谁可以对何种资源执行什么操作
    2. 容易混淆的概念
    3. 身份认证：确认你是谁
    4. 会话管理：维持后续的请求状态
    5. 访问控制：决定有什么权限
    6. 核心观点：认证通过不等于拥有权限。绝大多数漏洞发生在开发者完成了认证校验，却忘记了授权校验。

    访问控制的类型¶

    1. 垂直访问控制：不同角色之间的权限差异【管理员&普通用户】
    2. 水平访问控制：相同角色不同用户之间的数据隔离【用户A的订单&用户B的订单】
    3. 依赖上下文的访问控制：基于应用状态或操作顺序的限制【必须先提交订单才能结算，不能直接请求结算接口】

      2. 实战手法：

        1. 目录爆破：使用ffuf、dirsearch或Burp Intruder，字典包含/admin、/management、/console、/api/swagger

        2. 前端泄露：查看JS文件。搜索location.href、window.open或apiUrl，经常能发现未公开的管理接口或相关逻辑。

          1. 实战示例：通过script脚本泄露接口

          1. 检查robots.txt、sitemap.xml、.git/HEAD。
          2. 实战示例：通过robors.txt文件发现/administrator-panel接口
        4. 基于参数的访问控制
          1. 漏洞原因：服务端完全信任客户端传递的“权限标识”
          2. 实战手法

        5. 修改Cookie：抓包发现Cookie: role=user，改为role=admin或role=administrator

        6. 修改请求参数：URL中带有?isAdmin=false、?permission=view，直接改为true或edit

        7. 修改隐藏表单的值：HTML中<input type="hidden" name="userType" value="normal">，改为value="admin"。

        如何防御¶

        1. 默认拒绝原则 (Deny by Default)：
        2. 所有接口、方法、目录，除非明确声明为PUBLIC，否则一律视为PRIVATE。反例：先写允许逻辑，最后加拒绝逻辑。
        3. 单一的全局鉴权机制：
        4. 不要在每个Controller方法里写if(role==admin)。使用框架统一的拦截器（Interceptor）、中间件（Middleware）或过滤器（Filter）。
        5. 推荐：Spring Security @PreAuthorize、ASP.NET [Authorize] 属性、Express.js passport + 自定义中间件。
        6. 代码级声明：
          1. 权限必须与业务代码绑定，而不是写在配置文件中（配置文件容易遗忘）。使用注解是较好的实践： java @PreAuthorize("hasPermission(#userId, 'User', 'read')") public User getUser(String userId) { ... }
        7. 服务端强制校验 (永不信任前端)：
        8. 禁止依赖隐藏输入框、前端JS变量、不可见的菜单。
        9. 禁止依赖客户端传来的role、isAdmin参数。
        10. 必须从服务端的Session或JWT解析出当前用户ID，与请求的目标资源Owner ID进行比对。
