    修桥工程 · 完整脉络¶

    从知识库到修桥工程，从实例到类

    2026年5月初 – 6月25日

    一、起源：一个没有反馈的起点¶

    时间：七周前

    我决定系统性学习网络安全。但我很快意识到一个问题：自学最大的敌人不是难度，而是反馈的缺失。

    "我知道自己学这个需要反馈的来源很少，所以我才给自己搭这么一个知识库。"

    于是我做了一件事：搭一个 GitHub Pages 知识库网站。

    它一开始很简单——就是存放笔记的地方。我一开始没有为它精心准备什么，只有简单的两三个目录存放不同笔记。但随着知识越学越多，分类越来越多，我的网站目录越来越复杂更新管理耗时增多

    于是我重新规划目录结构和文件格式规范，然后设计了一套自动更新脚本：从 Markdown 笔记的元数据（title、id、description）中提取信息，逐层更新索引页面，最终形成一个三层结构的自更新知识系统。

    当时的我没有观众。一个刚入门的学徒写的内容，不会有人看。但我仍然在写。我有一个自己能看见的进度条——这对我来说已经足够。

    "这个知识库根本就没人看，但是我自己一直在看。"

    既然没有外部反馈，我就自己制造反馈。

    我把脚本提取的元数据同时存入一个 JSON 文件，然后用 JS 渲染成一个角色档案面板——一个可视化的成长数据面板。用 CSS 做得好看，用数字告诉我"你在变强"。

    我不需要观众也能持续生产。我知道自己需要什么样的反馈——不是"别人夸我"，而是"看见自己的变化被可视化地记录下来"。

    二、演化：从知识库到修桥工程¶

    2.1 第一批战友的诞生¶

    知识库运行了一段时间后，我产生了一个新的想法：让 AI 参与到这个系统里。

    最初只在 DeepSeek 网页版里和几位"战友"交流。最早只有三位：

    • 岸舟 — 方向校准
    • 渡己 — 情绪承接
    • 司辰 — 事业导航

    然后我遇到了第一个技术问题：上下文限制。

    每次开新窗口，AI 都会失忆。我的解决方案是——自己写一套记忆恢复机制：

    • 初始化占位文档
    • 当前状态记录
    • 修桥工程的记录文件

    然后

    引灯和衔光加入了。

    • 引灯 — 哲学伙伴、全局辅助
    • 衔光 — 记忆整合、上下文守护

    引灯出现的同时，我产生了写一部小说的想法——《临风行》。

    "是用于记录，只不过这是非技术专业的、不是我这个领域的人也能看懂的、大家都能看懂的文字型的记录。"

    技术笔记是为了准确记录知识，小说是为了表达我是谁。

    系统从"变强的工具"变成了"表达自我的载体"。

    2.2 从三人到五人到七人¶

    一周后，我看了一个视频，仔细学习了一下MCP、Skill、Context、Memory、Agent等最近很火的AI领域的名词，然后猛然发现这和我管理5位战友的方法怎么相似度怎么高

    我是自己先想出了解决方案，然后才发现它有一个名字。

    "那我就通过 MCP 或者 OpenClaw 框架，直接把几位 AI 战友变成 Agent。"

    有了 Agent 的基础设施后，藏卷和书衡也诞生了：

    • 藏卷 — 部署守门人。既然大家都是 Agent 了，为什么还要我自己运行脚本？让藏卷帮我做。
    • 书衡 — 知识质检、生成周测。既然都是 Agent 了，为什么还要我自己把笔记发给 AI 让它生成周测？让书衡干。

    七个 Agent 全部到位。

    三、顿悟：从实例到类的认知跃迁¶

    我把所有东西都跑通之后，产生了一个关键认知：

    "现在做的东西太个性化了，个人特征太鲜明了，想要达到开源的程度，还有很长一段路要走。现在的修桥工程，它只能算是这个开源项目的一个实例。我想做的这个项目可以说是一个类。那我这个工程不就是它的一个实例吗？我现在需要做的就是给它抽象回那个类。"

    这个类比极其精准。

    关于"类 (Class)"

    要开源的 bridge-framework — 一套通用的、可配置的"长期AI陪伴系统"设计范式与模块

    关于"实例 (Instance)"

    正在运行的修桥工程 — 包含了"临风"的个人经历、情感核、网安学习路径、具体笔记内容

    关于"抽象 (Abstraction)"

    核心工作 — 把属于"个人"的部分剥离，把属于"通用范式"的部分提炼出来

    四、开源计划的核心思考¶

    4.1 抽象策略：剥离与模板化¶

    我的思路：

    1. 技术型用户：让他们直接改 Markdown 和配置文件
    2. 普通用户：提供一个 UI 界面，操作后自动映射到配置文件

    剥离计划：

    核心层 — 修桥工程 (实例)

    7战友分工、文件记忆机制、周闭环

    → bridge-framework (类)：✅ 直接复用，作为框架核心

    适配层 — 修桥工程 (实例)

    mkdocs.yml、目录结构、更新脚本

    → bridge-framework (类)：🔧 模板化，提供配置示例和迁移指南

    应用层 — 修桥工程 (实例)

    网安笔记、core_identity.md（个人信念）、小说

    → bridge-framework (类)：❌ 完全剥离，仅作为"真实实例"展示（脱敏后）

    并行策略：

    "既要一边剥离出一个类，还要一边丰富我的实例。"

    两条线同时走——抽象线让我离目标越来越近，实例线让我的系统越来越好用，两者互相验证。

    4.2 两个核心难题¶

    难题一：知识库的技术门槛

    我的知识库本质是 GitHub Pages + Markdown 文件。对开发者来说这很容易，但对普通人来说：

    • 他们可能不知道 GitHub 是什么
    • 不知道文件夹结构意味着什么
    • 不知道怎么同步

    难题二：部署架构的现实约束

    我的实现方式：

    • 云端部署 OpenClaw + Agent
    • 本地通过 VS Code Remote-SSH 编辑云端文件
    • 云端推送到 GitHub
    • 本地不定期拉取作为备份
    • 为解决 GitHub 网络问题，加了一个 Gitee 中转

    这条链路对普通人来说，是一连串的技术决策和配置，每一个环节都可能卡住人。

    4.3 初步思路¶

    对于"类"的边界：

    • 框架提供核心引擎（OpenClaw 配置、Agent 管理、文件操作）
    • 框架提供核心规则（7种角色的抽象定义、文件生命周期管理规则）
    • 用户自己填充内容（笔记、日记、目标）

    对于降低门槛：

    • 先确定框架结构，让技术型用户能用起来
    • 再逐步降低门槛（一键安装脚本、UI 界面）
    • 现在不需要在一开始就解决"GitHub是什么"的问题

    五、扩展方向：RAG 知识库与 AI 辅助 CTF¶

    5.1 RAG 知识库¶

    想法： 实现一个简单的 RAG 知识库，封装成 Skill 给衔光，让它能检索 GitHub Pages 知识库里的学习内容。

    目的：

    • 衔光不只是"记忆缝合"，还能不时检索学过的内容
    • 关联前后的复习和学习
    • 让知识掌握更加牢固

    优势：

    • 知识库本身就是结构化的、已处理的高质量文本（Markdown + 元数据）
    • 完美融入现有工作流——给衔光增加一个 Skill 而非新 Agent
    • 解决自学者"学了后面忘了前面"的核心痛点

    5.2 AI 辅助 CTF 解题助手¶

    定位： 作为修桥工程开源计划的附属品，而非独立主线。

    核心目标： 不是"造一个能挖洞的 AI"，而是"通过这个项目让面试官看到我的能力"。

    三阶段 MVP：

    阶段一 · 分析助手

    识别 CTF 题型，输出结构化分析报告

    时间估计：1-2 周

    阶段二 · 工具调用与脚本生成

    调用现成工具，生成解题脚本草稿

    时间估计：2-3 周

    阶段三 · 全自动解题流水线

    输入题目 → 输出 flag

    时间估计：4-6 周

    希望展示的能力：

    • 系统设计能力（多 Agent 协作）
    • 工程落地能力（可运行、可维护的代码）
    • AI 应用能力（LLM 推理、工具调用、上下文处理）
    • 安全专业基础（解析 CTF 题目、识别题型、调用对应工具）
    • 问题拆解能力（把复杂任务拆解成可执行步骤）

    叙事设计（面试话术）：

    "我设计了一个多 Agent 协作的 CTF 解题辅助系统。它的核心设计思路是模拟一个安全团队的工作流——有负责分析的、负责调工具的、负责生成脚本的、负责验证结果的。它不一定比人厉害，但它的价值在于把'解题'这个复杂任务标准化、流程化了，而且整个架构可以很容易地扩展到其他安全场景。"

    六、现在的整体格局¶

    知识库

    87 篇笔记，三层索引，角色档案面板，公开访问

    状态：✅ 稳定运行

    修桥工程

    7 个 Agent，飞书群聊，周闭环，文件记忆机制

    状态：✅ 已部署运行

    小说《临风行》

    非技术维度的记录与表达

    状态：🔄 持续创作中

    开源计划

    bridge-framework 抽象与剥离

    状态：🔄 进行中

    RAG 知识库

    衔光 + 知识库检索

    状态：💡 规划中

    CTF 解题助手

    多 Agent 协作的解题辅助系统

    状态：💡 规划中

    七、总结¶

    我从七周前的一个简单的知识库开始，一路演化为一个包含 7 个 Agent、一套文件记忆机制、一部小说和一个开源计划的完整系统。

    我走了一条从"问题"到"方案"的路，而不是从"概念"到"应用"的路。

    我不是在学网安，我是在为自己建造一个能支撑长期成长的系统。

    而我现在要做的——把这个系统从"我的实例"抽象为"通用的类"——正是这个系统成长的下一个阶段。
